## Skill: Зафиксировать изменения в документации (ADR + sitemap)

### Когда использовать

- Добавляешь/меняешь маршруты, guards, роли.
- Принимаешь архитектурное решение, которое будут обсуждать через неделю.

### Шаги: sitemap

1. Обнови `docs/sitemap.md`:
   - путь,
   - lib/компонент,
   - guard,
   - описание.
2. Проверь, что маршруты покрывают все роли и use cases (минимум — ключевые).

### Шаги: ADR

1. Создай файл `docs/adr/ADR-NNN.md` по шаблону из `docs/DEVELOPMENT_PLAN.md`.
2. Заполни:
   - контекст,
   - варианты,
   - принятое решение,
   - последствия.

### Готово когда

- `docs/sitemap.md` соответствует текущим маршрутам.
- ADR описывает решение так, что его можно пересмотреть/оспорить без “контекста из головы”.

### Источник в репозитории

- `docs/DEVELOPMENT_PLAN.md` (E.3 sitemap, E.4 ADR)
