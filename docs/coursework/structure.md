# Структура курсовой работы по проекту Notary

Документ задает общую структуру курсовой работы. Конкретную тему, зону ответственности, перечень экранов и особенности компонента студент берет из своего индивидуального файла в каталоге `issues-course-work/`.

Курсовая должна показывать не только готовый интерфейс, но и связь компонента с архитектурой проекта: развертыванием, моделью данных, API, бизнес-правилами, ролями доступа и пользовательскими сценариями.

## 1. Что подготовить перед написанием

Перед оформлением курсовой нужно открыть:

- индивидуальное задание в `issues-course-work/`;
- `docs/Responsobility.md` - зона ответственности и роль пользователя;
- `docs/entities.md` - сущности и поля базы данных;
- `README.md` - запуск проекта и общая архитектура;
- `apps/web/DOCKER.md`, `apps/web/Dockerfile`, `docker-compose.yaml` - материалы по развертыванию;
- при наличии готовой реализации - файлы своего компонента в `apps/` и `libs/`.

В начале работы студент фиксирует:

| Что определить | Что указать в курсовой |
| --- | --- |
| Тема компонента | Название из индивидуального задания |
| Пользовательская роль | Гость, заявитель, нотариус, администратор |
| Основные функции | 3-7 ключевых функций из `docs/Responsobility.md` |
| Сущности БД | Сущности из `docs/entities.md`, которые использует компонент |
| API | Методы, через которые UI получает или меняет данные |
| UI-экраны | Список экранов, форм, таблиц, карточек, модальных окон |
| Ссылки на код | Пути к frontend, backend, contract/proto, Prisma-схеме |

## 2. Рекомендуемый объем и состав

Итоговый объем основной части: примерно 11-15 страниц без титульного листа, содержания, списка литературы и приложений.

| Раздел | Примерный объем | Что должно быть внутри |
| --- | ---: | --- |
| Введение | 1 стр. | Цель работы, актуальность, объект и предмет разработки, задачи |
| 1. Модель развертывания системы | 2 стр. | UML deployment diagram, описание ВМ, Docker, nginx, БД, хранилища, мониторинга |
| 2. Проектирование компонента | 4-5 стр. | Назначение компонента, UML class diagram, UML component diagram, API, бизнес-правила |
| 3. UI компонента и руководство пользователя | 5-6 стр. | Экраны, состояния, скриншоты, сценарии, валидация, роли доступа |
| Заключение | 1 стр. | Что реализовано, что проверено, ограничения и развитие |
| Приложения | по необходимости | Полные диаграммы, листинги, дополнительные скриншоты |

## 3. Введение

Во введении нужно кратко объяснить, какую часть системы разрабатывает студент и зачем она нужна пользователям.

Обязательные пункты:

- **Актуальность:** какую проблему решает компонент в системе нотариальной оценки наследственного имущества.
- **Цель:** разработать и описать компонент из индивидуального задания.
- **Объект:** веб-система Notary для работы с заявками, оценкой, документами, платежами и администрированием.
- **Предмет:** конкретный UI/API/модуль, закрепленный за студентом.
- **Задачи:** изучить архитектуру, описать развертывание, спроектировать модель данных, описать API, реализовать/показать UI, подготовить руководство пользователя.

Пример формулировки цели:

> Цель работы - разработать и описать компонент "Личный кабинет заявителя", обеспечивающий подачу заявок на оценку наследственного имущества, загрузку документов и отслеживание статуса обработки.

## 4. Раздел 1. Модель развертывания системы

Раздел общий для всех студентов, но на диаграмме нужно выделить свой компонент или пользовательский сценарий.

### 4.1. Диаграмма развертывания UML

На диаграмме нужно показать:

- браузер пользователя и Angular-приложение;
- nginx/reverse proxy;
- API-сервер NestJS;
- PostgreSQL;
- файловое хранилище, например MinIO/S3;
- мониторинг и логи: Prometheus, Grafana, Loki, Promtail, если используются в описываемом контуре;
- внешние сервисы, если они нужны компоненту: ЮKassa, SMTP, OAuth-провайдеры, геокодирование, аналитика;
- порты и протоколы: HTTP/HTTPS, Connect/gRPC/REST, PostgreSQL, S3 API;
- границу виртуального сервера и внешние сервисы за его пределами.

Минимальный состав диаграммы:

```text
Пользовательский браузер
  -> nginx / reverse proxy : HTTPS 443
  -> Angular frontend
  -> NestJS API : HTTP/Connect
  -> PostgreSQL : 5432
  -> MinIO/S3 : файлы и документы
  -> Grafana/Loki/Prometheus : мониторинг и логи
```

Для индивидуальной части нужно добавить пометку, где находится компонент:

- frontend-экран в Angular;
- backend-сервис или контроллер;
- таблицы БД;
- внешние сервисы, если они участвуют в сценарии.

### 4.2. Описание развертывания на виртуальном сервере

В тексте нужно описать:

1. **Сервер и окружение:** Linux VM, Docker, Docker Compose, переменные окружения.
2. **Контейнеры:** frontend/nginx, API, PostgreSQL, MinIO/S3, Grafana/Prometheus/Loki при наличии.
3. **Сеть:** какие контейнеры доступны извне, какие работают только во внутренней сети.
4. **Порты:** 80/443 для nginx, 3000 для API, 5432 для PostgreSQL, 3001 для Grafana при локальной разработке.
5. **Хранение данных:** volume для PostgreSQL, volume/bucket для файлов документов и отчетов.
6. **Безопасность:** HTTPS, секреты в `.env`, JWT/сессии, ограничение доступа к админским маршрутам.
7. **Резервное копирование:** дампы PostgreSQL и копирование файлового хранилища.
8. **Мониторинг:** метрики, логи контейнеров, просмотр ошибок API.

В конце подраздела нужно добавить 2-4 ссылки на проектные файлы, например:

- `docker-compose.yaml`;
- `apps/web/Dockerfile`;
- `apps/web/DOCKER.md`;
- `README.md`;
- `monitoring/grafana/provisioning/dashboards/`.

## 5. Раздел 2. Проектирование компонента

Раздел индивидуальный. Его содержание берется из файла студента в `issues-course-work/` и уточняется по документации проекта.

### 5.1. Назначение и границы компонента

Нужно ответить на вопросы:

- какую задачу пользователя решает компонент;
- для какой роли он доступен;
- какие действия пользователь может выполнить;
- какие действия запрещены;
- какие данные компонент читает и изменяет;
- с какими соседними компонентами он связан.

Рекомендуемая таблица:

| Элемент | Описание |
| --- | --- |
| Роль пользователя | Например: заявитель, нотариус, администратор |
| Основной сценарий | Что пользователь делает от начала до результата |
| Входные данные | Формы, фильтры, файлы, параметры |
| Выходные данные | Заявка, отчет, платеж, уведомление, запись аудита |
| Соседние модули | Авторизация, файлы, платежи, уведомления, аудит |
| Ограничения | Права доступа, лимиты, статусы, валидация |

### 5.2. Диаграмма классов UML

Диаграмма классов должна строиться на основе `docs/entities.md`. Для своего компонента нужно показать не всю базу, а только связанные сущности.

Обязательные требования:

- указать поля сущностей, которые используются в компоненте;
- отметить первичные и внешние ключи;
- показать связи 1:1, 1:N, N:M;
- добавить enum-значения для статусов и ролей;
- отдельно показать концептуальные классы, если они нужны, но пока не описаны в `entities.md`.

Примеры выбора сущностей:

| Тип задания | Основные сущности |
| --- | --- |
| Регистрация и вход | `User`, концептуально `PasswordResetToken`, OAuth profile |
| Личный кабинет заявителя | `User`, `Assessment`, `RealEstateObject`, `Document`, `Notification` |
| Личный кабинет нотариуса | `User`, `Assessment`, `Document`, `AssessmentReport`, `Subscription` |
| Админ-панель | `User`, `Assessment`, `Payment`, `AuditLog`, `TariffPlan` |
| Оценка недвижимости | `Assessment`, `RealEstateObject`, `Document`, `RealEstateAppraisalResult`, `AssessmentReport` |
| Платежи | `Payment`, `TariffPlan`, `Subscription`, `Promo`, `Sale` |
| Рассылка | `Mailing`, `SmtpClient`, `SubscriberGroup`, `User` |
| Уведомления | `Notification`, `User`, `AuditLog` |
| Аудит и логи | `AuditLog`, `User`, `Assessment` |
| Справочный раздел / FAQ | концептуально `Article`, `Category`, `User` |
| Чат поддержки | концептуально `SupportTicket`, `Message`, `Attachment`, `User` |

### 5.3. Диаграмма компонентов UML

Диаграмма компонентов показывает, как UI связан с API, сервисами и хранилищами.

На диаграмме нужно отразить:

- Angular feature/component;
- route или страница, на которой расположен UI;
- API-клиент или contract/proto;
- backend controller/RPC service;
- application service;
- repository/Prisma;
- PostgreSQL;
- файловое хранилище;
- внешние сервисы;
- сервис уведомлений или аудита, если компонент создает события.

Типовая цепочка:

```text
Angular UI -> API client -> NestJS controller/RPC service -> Service -> Repository/Prisma -> PostgreSQL
```

Для файлов:

```text
Angular upload form -> API upload endpoint -> Storage service -> MinIO/S3
```

Для платежей:

```text
Angular checkout -> Billing API -> Payment service -> ЮKassa -> Webhook -> Payment status update
```

### 5.4. API и контракты

Нужно описать API, который требуется компоненту. Если точные endpoints еще не реализованы, допускается указать проектируемые методы, но нужно явно написать, что это проектируемый контракт.

Рекомендуемая таблица:

| Метод | Endpoint / RPC | Назначение | Входные параметры | Ответ | Ошибки |
| --- | --- | --- | --- | --- | --- |
| GET | `/api/example` | Получить список | `page`, `limit`, `filter` | `Example[]` | `401`, `403` |
| POST | `/api/example` | Создать запись | `CreateExampleDto` | `Example` | `400`, `409` |

Для каждого метода нужно указать:

- кто имеет доступ;
- какие поля обязательны;
- какие статусы или ошибки возможны;
- какая сущность БД создается или изменяется;
- где в коде расположен обработчик или где он должен быть расположен.

### 5.5. Бизнес-правила и ограничения доступа

Этот подраздел связывает UI, API и роли.

Минимум нужно описать:

- какие роли видят экран;
- какие роли могут создавать, редактировать, удалять записи;
- какие статусы доступны для изменения;
- какие действия требуют подтверждения;
- какие действия логируются;
- какие данные пользователь видит только по своим заявкам;
- какие ошибки показываются пользователю.

Пример:

| Правило | Описание |
| --- | --- |
| Доступ к списку заявок | Заявитель видит только свои заявки, администратор видит все |
| Изменение статуса | Статус может менять нотариус или администратор |
| Загрузка файлов | Разрешены PDF/JPG/PNG, размер ограничен настройками сервера |
| Аудит | Создание, изменение и удаление записываются в `AuditLog` |

## 6. Раздел 3. UI компонента и руководство пользователя

Раздел должен подтверждать, что компонент можно понять и использовать без чтения кода.

### 6.1. Перечень экранов и состояний

Нужно перечислить все экраны и состояния интерфейса.

Для каждого экрана указать:

- URL или маршрут;
- роль пользователя;
- назначение экрана;
- основные элементы;
- данные, которые загружаются;
- действия пользователя;
- результат действия.

Рекомендуемая таблица:

| Экран | Маршрут | Роль | Основные элементы | Действия |
| --- | --- | --- | --- | --- |
| Список заявок | `/applicant/assessments` | Заявитель | Таблица, фильтры, статусы | открыть, создать, отфильтровать |
| Карточка заявки | `/applicant/assessments/:id` | Заявитель | Данные объекта, документы, история | загрузить файл, посмотреть статус |

Нужно показать состояния:

- загрузка данных;
- пустой список;
- данные успешно загружены;
- ошибка загрузки;
- ошибка валидации;
- действие выполнено успешно;
- нет прав доступа.

### 6.2. Формы, таблицы, фильтры и валидация

Если в компоненте есть формы, нужно описать:

- поля формы;
- обязательность;
- формат данных;
- ограничения длины/размера;
- сообщения об ошибках;
- поведение кнопок сохранения/отмены.

Если есть таблица или список, нужно описать:

- колонки;
- сортировку;
- фильтры;
- поиск;
- пагинацию;
- массовые действия, если есть;
- переход к деталям.

Пример таблицы для формы:

| Поле | Тип | Обязательное | Валидация | Ошибка |
| --- | --- | --- | --- | --- |
| Email | string | да | формат email | Некорректный email |
| Телефон | string | да | российский номер | Укажите телефон в формате +7... |
| Файл | file | зависит от сценария | PDF/JPG/PNG, лимит размера | Недопустимый формат файла |

### 6.3. Скриншоты

В курсовую нужно вставить минимум 3 скриншота, а для сложных компонентов - 5-7.

Обязательные типы скриншотов:

- основной экран с данными;
- форма создания или редактирования;
- карточка/детальный просмотр;
- состояние ошибки или валидации;
- состояние пустого списка или загрузки;
- модальное окно подтверждения, если оно есть.

Под каждым скриншотом нужно написать 2-4 предложения:

- что изображено;
- какие элементы важны;
- какое действие выполняет пользователь;
- какой результат ожидается.

### 6.4. Руководство пользователя

Руководство пишется пошагово для основной роли из индивидуального задания.

Формат сценария:

1. Пользователь открывает страницу.
2. Система загружает данные.
3. Пользователь заполняет поля или выбирает фильтры.
4. Пользователь нажимает основную кнопку действия.
5. Система проверяет данные.
6. Система сохраняет результат и показывает уведомление.
7. Пользователь видит обновленное состояние.

Для каждого компонента нужно описать минимум 3 сценария:

- основной успешный сценарий;
- сценарий с ошибкой или некорректными данными;
- сценарий просмотра результата или истории.

## 7. Заключение

- какой компонент был разработан или описан;
- какие функции реализованы;
- какие диаграммы и проектные материалы подготовлены;
- как компонент связан с общей архитектурой Notary;

- какие ограничения остались;
- как компонент можно развивать дальше.

Пример направлений развития:

- расширение фильтров и поиска;

- улучшение уведомлений;
- интеграция с внешним сервисом;
- повышение безопасности и аудита;
- адаптация под мобильные устройства.

## 9. Как использовать индивидуальные задания

Каждый файл в `issues-course-work/` задает персональную тему. Его не нужно копировать в курсовую целиком. Из него нужно перенести и раскрыть:

- ФИО и название компонента;
- зону ответственности;
- список сущностей для диаграммы классов;
- требуемые UI-экраны;
- бизнес-правила;
- API-методы;
- критерии готовности.

Если индивидуальный файл короткий, его нужно расширить в самой курсовой по правилам из этого документа:

- взять функции из `docs/Responsobility.md`;
- взять поля сущностей из `docs/entities.md`;
- добавить 5-7 API-методов;
- описать минимум 3 UI-сценария;
- добавить 3-7 скриншотов;
- указать ссылки на код.
