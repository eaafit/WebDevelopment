# Введение

Разработка web-приложения для центра Нотариуса.

Актуальность работы обусловлена тем, что в веб-приложениях, особенно в сфере оказания нотариальных и юридических услуг, важным компонентом является подсистема управления тарификацией и скидочными программами. Администратору необходимы инструменты для создания, редактирования и контроля тарифных планов подписок, управления скидками и промокодами, которые влияют на расчёт стоимости услуг нотариуса. Разрабатываемый модуль управления тарифными планами, скидками и промокодами в рамках проекта «WebDevelopment» является актуальным, поскольку обеспечивает централизованное администрирование ценовой политики, поддерживает гибкую систему скидок и промокодов с ограничениями по сроку действия и количеству использований.

Цель работы – реализация административного модуля управления тарифными планами, скидками и промокодами, включающего создание, редактирование, просмотр и удаление записей, а также интеграцию с модулями платежей и подписок.

Задачи работы:

1. Модель развертывания системы – диаграмма развертывания UML и описание контейнеризации с использованием Docker Compose, переменных окружения, портов, хранения данных и мониторинга.
2. Компонент тарификации – диаграммы классов и компонентов UML, спецификация API-методов (CRUD для тарифных планов, скидок и промокодов) с форматами данных и ссылками на существующий код.
3. UI компонента – требования к интерфейсу, бизнес-правила для роли администратора, скриншоты экранов и подробное руководство пользователя по созданию, редактированию и удалению тарифных планов, скидок и промокодов.

## 1.2. Описание развертывания на виртуальном сервере

В данном подразделе рассмотрены пять аспектов развертывания системы на виртуальном сервере: контейнеризация, переменные окружения, порты и сеть, хранение данных, мониторинг.

1. Контейнеризация:

- Проект использует Docker Compose (файл `docker-compose.yaml`) для оркестрации всех сервисов.
- Каждый компонент (API, фронтенд, БД, MinIO, Nginx) работает в изолированном контейнере.
- Сборка фронтенда осуществляется с помощью многоэтапного Dockerfile (`apps/web/Dockerfile`): на первом этапе Node.js собирает Angular-приложение, на втором – Nginx раздаёт статику.

2. Переменные окружения (ключевые для модуля тарификации):

- `DATABASE_URL` – строка подключения к PostgreSQL.
- `PAYMENT_PROVIDER` – выбор платёжного провайдера (robokassa / yookassa).
- `YOOKASSA_SHOP_ID`, `YOOKASSA_SECRET_KEY` – параметры доступа к API ЮKassa.
- `ROBOKASSA_MERCHANT_LOGIN`, `ROBOKASSA_PASSWORD_1`, `ROBOKASSA_PASSWORD_2` – параметры доступа к API Robokassa.
- `S3_ENDPOINT`, `S3_ACCESS_KEY`, `S3_SECRET_KEY` – параметры доступа к MinIO.
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS` – настройки отправки уведомлений.

3. Порты и сеть:

- Внешний доступ: порты 80 (HTTP с редиректом на 443) и 443 (HTTPS) через Nginx.
- Внутренняя сеть Docker: API (3000), PostgreSQL (5432), MinIO (9000), Prometheus (9090), Grafana (3001).
- Все межконтейнерные взаимодействия осуществляются по внутренней Docker-сети без экспозиции портов наружу.

4. Хранение данных:

- PostgreSQL: используется Docker volume `pgdata` для сохранения базы данных между перезапусками.
- MinIO: используется volume `minio_data` для хранения чеков, документов и отчётов.
- Резервное копирование: ежедневные дампы PostgreSQL (`pg_dump`) и синхронизация данных MinIO в резервное S3-хранилище.

5. Мониторинг:

- Prometheus собирает метрики с API (количество запросов, задержки, ошибки) и бизнес-метрики (количество созданных/удалённых тарифных планов, применённых промокодов).
- Grafana предоставляет дашборды для визуализации: активность администраторов, статистика по тарифным планам, частота использования промокодов.
- Логирование реализовано через структурированные логи с контекстом для трассировки действий администратора.

# Глава 2. Компонент управления тарифными планами, скидками и промокодами

## 2.1. Диаграмма классов UML

Начнем с демонстрации диаграммы классов UML (См. рис. 2.1).

**Подробное описание для построения диаграммы классов:**

**Общая структура:** Диаграмма классов показывает сущности базы данных, их атрибуты и связи. Располагается горизонтально. В центре — Payment (связующая сущность), вокруг него — остальные классы.

**Шаг 1 — Расположение классов (слева направо, сверху вниз):**

**Класс 1: User** (левый верхний угол)

- Прямоугольник с тремя секциями (заголовок, атрибуты, методы — методы можно не заполнять)
- Заголовок: «User» (темно-синий фон, белый текст)
- Атрибуты (все с «+» в начале строки):
  - «+ Id: UUID (PK)»
  - «+ Email: varchar (unique)»
  - «+ PasswordHash: varchar»
  - «+ FullName: varchar»
  - «+ Role: UserRole»
  - «+ PhoneNumber: varchar?» (вопросительный знак = nullable)
  - «+ IsActive: boolean»
  - «+ CreatedAt: timestamp»
  - «+ UpdatedAt: timestamp»

**Класс 2: TariffPlan** (правый верхний угол, над Promocode)

- Заголовок: «TariffPlan» (темно-синий фон)
- Атрибуты (все с «+»):
  - «+ Id: Int (PK)» — целое число, Primary Key
  - «+ Name: String»
  - «+ Price: Float»
  - «+ Description: String?» — nullable
  - «+ IsActive: Boolean = true» — default значение
  - «+ ValidFrom: DateTime»
  - «+ ValidTo: DateTime»
  - «+ CreatedAt: DateTime = now()»
  - «+ UpdatedAt: DateTime = updatedAt»

**Класс 3: Discount** (правый верхний угол, рядом с TariffPlan)

- Заголовок: «Discount» (темно-синий фон)
- Атрибуты:
  - «+ Id: Int (PK)»
  - «+ Name: String»
  - «+ Percentage: Float»
  - «+ Description: String?»
  - «+ IsActive: Boolean = true»
  - «+ ValidFrom: DateTime»
  - «+ ValidTo: DateTime»
  - «+ MinOrderAmount: Float?»
  - «+ MaxDiscountAmount: Float?»
  - «+ CreatedAt: DateTime = now()»
  - «+ UpdatedAt: DateTime = updatedAt»

**Класс 4: Promocode** (правый средний уровень)

- Заголовок: «Promocode» (темно-синий фон)
- Атрибуты:
  - «+ Id: Int (PK)»
  - «+ Code: String (unique)» — уникальный код
  - «+ DiscountType: enum» — тип скидки (percentage или fixed)
  - «+ DiscountValue: Float» — значение скидки
  - «+ MaxUses: Int» — максимум использований
  - «+ UsedCount: Int = 0» — счетчик использований
  - «+ Description: String?»
  - «+ IsActive: Boolean = true»
  - «+ ValidFrom: DateTime»
  - «+ ValidTo: DateTime»
  - «+ CreatedAt: DateTime = now()»
  - «+ UpdatedAt: DateTime = updatedAt»

**Класс 5: Payment** (центр диаграммы) — связующий класс

- Заголовок: «Payment» (темно-синий фон)
- Атрибуты (ключевые для связей):

  - «+ UserId: UUID (FK)» — Foreign Key к User
  - «+ Type: PaymentType» — enum (см. ниже)
  - «+ SubscriptionId: UUID? (FK)» — nullable FK
  - «+ AssessmentId: UUID? (FK)» — nullable FK
  - «+ PromoId: UUID? (FK)» — Foreign Key к Promocode
  - «+ Amount: Decimal(15,2)» — сумма платежа
  - «+ DiscountAmount: Decimal?» — сумма скидки
  - «+ PaymentDate: DateTime = now()»
  - «+ Status: PaymentStatus» — enum
  - «+ PaymentMethod: String?»
  - «+ TransactionId: String? (unique)»
  - «+ AttachmentFileName: String?»
  - «+ AttachmentFileUrl: String?»
  - «+ ReceiptStatus: PaymentReceiptStatus?» — enum

**Класс 6: Subscription** (левый нижний угол)

- Заголовок: «Subscription» (темно-синий фон)
- Атрибуты:
  - «+ Id: UUID (PK)»
  - «+ UserId: UUID (FK)»
  - «+ Plan: SubscriptionPlan» — enum (Basic, Premium, Enterprise)
  - «+ StartDate: Date»
  - «+ EndDate: Date»
  - «+ IsActive: Boolean»

**Класс 7: Assessment** (левый нижний, под User)

- Заголовок: «Assessment» (темно-синий фон)
- Атрибуты (ключевые):
  - «+ Id: UUID (PK)»
  - «+ UserId: UUID (FK)»
  - «+ Status: AssessmentStatus» — enum
  - «+ Address: String»
  - «+ Description: String?»
  - «+ EstimatedValue: String?»
  - «+ CreatedAt: timestamp»
  - «+ UpdatedAt: timestamp»

**Шаг 2 — Enum-классы (перечисления):**

**PaymentType** (<<enumeration>>)

- Прямоугольник с заголовком «<<enumeration>> PaymentType»
- Значения (в столбик):
  - «Subscription»
  - «Assessment»
  - «DocumentCopy»
- Цвет: светло-голубой или серый
- Связь: пунктирная стрелка с треугольником (зависимость) от Payment к PaymentType

**PaymentStatus** (<<enumeration>>)

- Заголовок: «<<enumeration>> PaymentStatus»
- Значения:
  - «Pending»
  - «Completed»
  - «Failed»
  - «Refunded»
- Связь: пунктирная стрелка от Payment

**PaymentReceiptStatus** (<<enumeration>>)

- Заголовок: «<<enumeration>> PaymentReceiptStatus»
- Значения:
  - «Pending»
  - «Available»
  - «Failed»
- Связь: пунктирная стрелка от Payment

**Шаг 3 — Связи между классами (стрелки):**

1. **User — Payment**:
   - Тип связи: 1:N (один ко многим)
   - От User (1) к Payment (N)
   - Стрелка: сплошная линия с ромбиком на стороне User (агрегация) или просто ассоциация
   - Подпись связи: «UserId (FK)»

2. **TariffPlan — Subscription**:
   - Тип связи: 1:N
   - От TariffPlan (1) к Subscription (много)
   - Стрелка из TariffPlan в Subscription
   - Подпись: связь через поле Plan в Subscription

3. **Payment — Subscription** (опциональная):
   - Тип: 0..1 (ноль или один)
   - От Payment к Subscription
   - Подпись: «SubscriptionId (FK)»

4. **Payment — Assessment** (опциональная):
   - Тип: 0..1
   - От Payment к Assessment
   - Подпись: «AssessmentId (FK)»

5. **Promocode — Payment**:
   - Тип: 0..1 (один промокод может быть использован в многих платежах)
   - От Promocode к Payment
   - Подпись: «PromoId (FK)»

Отобразим поток данных при создании тарифного плана:

1. Администратор заполняет форму в `PlanComponent`
2. `TariffPlanService.create()` (Angular) отправляет POST-запрос на `/api/tariff-plans`
3. `TariffPlanController` вызывает `TariffPlanService.create()` (NestJS)
4. `TariffPlanService` создаёт запись в PostgreSQL через `PrismaService`
5. Ответ возвращается фронтенду, список тарифов обновляется
6. Администратор видит новый тарифный план в таблице

Поток данных при удалении промокода:

1. Администратор нажимает «Удалить» → открывается модальное окно подтверждения
2. После подтверждения вызывается `PromocodeService.delete()` (Angular)
3. `PromocodeController.remove()` вызывает `PromocodeService.remove()` (NestJS)
4. `PrismaService` удаляет запись из PostgreSQL
5. Список промокодов обновляется, промокод исчезает из таблицы

## 3.2. Бизнес-правила

Для роли Администратор:

- Может создавать, редактировать и удалять тарифные планы, скидки и промокоды
- Может деактивировать тарифный план (переключить флаг `isActive`)
- Может активировать/деактивировать скидку и промокод
- При создании промокода: код должен быть уникальным, при нарушении — ошибка 409 Conflict
- При редактировании промокода: если код изменён на уже существующий — ошибка 409 Conflict
- Удаление — жёсткое, без возможности восстановления

Ограничения доступа:

- Просмотр / Создание / Редактирование / Удаление – только Администратор
- Доступ к маршрутам `/admin/plans`, `/admin/discounts`, `/admin/promocodes` защищён `roleGuard(UserRole.Admin)`

Валидация данных:

- Название тарифного плана / скидки / промокода — обязательно
- Цена тарифного плана — обязательна, > 0
- Процент скидки — обязателен, > 0
- Код промокода — обязателен, уникален
- Даты действия (validFrom, validTo) — обязательны
- Максимальное количество использований промокода — обязательно, > 0

| Правило                       | Описание                                                                                 |
| ----------------------------- | ---------------------------------------------------------------------------------------- |
| Доступ к управлению тарифами  | Только администратор может создавать, редактировать и удалять тарифные планы             |
| Уникальность промокода        | Код промокода должен быть уникальным; при нарушении — HTTP 409                           |
| Период действия               | validFrom и validTo обязательны; тариф/скидка/промокод активен только в указанный период |
| Деактивация                   | Администратор может деактивировать тарифный план без удаления                            |
| Удаление                      | Удаление без восстановления (hard delete); требуется подтверждение в модальном окне      |
| Лимит использований промокода | Промокод перестаёт действовать после достижения maxUses                                  |

## 3.4. Руководство пользователя

**Создание тарифного плана**

1. Действие: нажать кнопку «Создать тарифный план»
   - В системе: открывается модальное окно с пустой формой
   - Результат: отображается форма с обязательными полями (Название, Цена, Даты действия)
2. Действие: заполнить поля Название, Цена, Описание, Даты действия, Флаг активности
   - В системе: выполняется валидация в реальном времени, ошибки отображаются под полями
   - Результат: форма готова к отправке
3. Действие: нажать «Сохранить»
   - В системе: `TariffPlanService.create()` → POST `/api/tariff-plans` → PostgreSQL
   - Результат: тарифный план создан, список обновлён, модальное окно закрывается

**Редактирование тарифного плана**

1. Действие: в списке тарифных планов нажать «Редактировать»
   - В системе: открывается модальное окно с предзаполненными данными
   - Результат: форма открыта, поля заполнены текущими значениями
2. Действие: изменить нужные поля (цена, описание, статус и т.д.)
   - В системе: выполняется валидация в реальном времени
   - Результат: форма готова к отправке
3. Действие: нажать «Сохранить»
   - В системе: `TariffPlanService.update()` → PUT `/api/tariff-plans/:id` → PostgreSQL
   - Результат: тарифный план обновлён, список обновлён

**Деактивация тарифного плана**

1. Действие: в списке тарифных планов нажать «Деактивировать» (для активного плана)
   - В системе: флаг `isActive` переключается на `false`
   - Результат: тарифный план помечен как неактивный, статус в таблице обновлён

**Создание промокода**

1. Действие: нажать кнопку «Создать промокод»
   - В системе: открывается модальное окно с пустой формой
   - Результат: отображается форма с обязательными полями
2. Действие: заполнить поля Код, Тип скидки, Значение скидки, Лимит использований, Даты действия
   - В системе: выполняется валидация, проверка уникальности кода
   - Результат: форма готова к отправке
3. Действие: нажать «Сохранить»
   - В системе: `PromocodeService.create()` → POST `/api/promocodes` → PostgreSQL
   - Результат: промокод создан, список обновлён. При дублировании кода — ошибка 409 Conflict

**Удаление промокода**

1. Действие: в списке промокодов нажать «Удалить»
   - В системе: открывается модальное окно подтверждения с кодом промокода
   - Результат: отображается предупреждение «Это действие нельзя отменить»
2. Действие: проверить данные и нажать «Удалить»
   - В системе: `PromocodeService.delete()` → DELETE `/api/promocodes/:id` → PostgreSQL
   - Результат: промокод удалён без возможности восстановления, список обновлён

**Сценарии использования**

- Сценарий 1: Создание нового тарифного плана «Премиум»
  Администратор создаёт тарифный план с названием «Премиум», ценой 5000 руб., периодом действия 1 год и статусом «Активен».

- Сценарий 2: Применение промокода с ограничением
  Администратор создаёт промокод «NEWYEAR2026» с типом скидки «процент», значением 15%, лимитом 100 использований и сроком действия до 31.01.2026.

- Сценарий 3: Удаление устаревшей скидки
  Администратор удаляет скидку, срок действия которой истёк, через модальное окно подтверждения.
