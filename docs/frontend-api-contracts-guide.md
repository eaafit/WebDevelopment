# Лабораторная работа. Вызов методов API через TypeScript (Connect RPC)

**Дисциплина:** Web-разработка  
**Тема:** Работа с API-контрактами на фронтенде. TypeScript, Connect RPC, типизация запросов и ответов.

---

## 1. Цель работы

- Научиться создавать RPC-клиент для вызова методов API на основе типизированных контрактов.
- Освоить работу с типами из `@notary-portal/api-contracts`: запросы, ответы, enum, даты (Timestamp).
- Закрепить практику выделения фасад-сервиса и маппинга DTO → UI-модели.

---

## 2. Краткие теоретические сведения

API проекта построен на **Connect RPC** — протоколе поверх HTTP. Контракты (сервисы, сообщения, перечисления) генерируются из `.proto` файлов и поставляются пакетом **`@notary-portal/api-contracts`**. На фронтенде используются только эти типы; вручную описывать структуры запросов/ответов не требуется.

**Основные зависимости:**

| Пакет                          | Назначение                                                                                                 |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| `@notary-portal/api-contracts` | Сервисы и типы (Request/Response, enum, message)                                                           |
| `@connectrpc/connect`          | Функция `createClient`                                                                                     |
| `@connectrpc/connect-web`      | `createConnectTransport` для браузера                                                                      |
| `@bufbuild/protobuf`           | Утилиты для `google.protobuf.Timestamp` (`timestampDate`, `timestampFromDate` из `@bufbuild/protobuf/wkt`) |

Эталонный пример в проекте: [transactions-api.service.ts](../libs/web/notary/src/lib/features/dashboard/transactions/transactions-api.service.ts).

## 5. Ход работы

### Шаг 1. Создание RPC-клиента

**Задание:** Создать клиент для одного из сервисов API (например, `PaymentService`), используя Connect RPC.

**Порядок выполнения:**

1. В своём сервисе (или в новом файле для тренировки) добавьте импорты:
   - из `@connectrpc/connect` — `createClient`;
   - из `@connectrpc/connect-web` — `createConnectTransport`;
   - из `@notary-portal/api-contracts` — нужный сервис (например, `PaymentService`).

2. Определите базовый URL для API. В проекте используется логика:
   - среда без `window` (SSR) → `http://localhost:3000`;
   - браузер, localhost с портом ≠ 3000 → `http://<hostname>:3000`;
   - иначе → `window.location.origin`.  
     Реализуйте функцию `buildRpcBaseUrl()` по этой логике или возьмите её из эталонного файла.

3. Создайте клиент:
   ```typescript
   const client = createClient(
     PaymentService,
     createConnectTransport({ baseUrl: buildRpcBaseUrl() }),
   );
   ```
   Рекомендуется хранить клиент в поле класса (например, Angular-сервиса) и не создавать его заново при каждом вызове.

**Контроль:** Клиент создаётся без ошибок TypeScript; типы для `PaymentService` подставляются автоматически.

---

### Шаг 2. Вызов RPC-метода и типизация запроса/ответа

**Задание:** Вызвать один из методов сервиса (например, `getPaymentHistory`), используя только типы из `@notary-portal/api-contracts`.

**Порядок выполнения:**

1. Импортируйте типы запроса и ответа из `@notary-portal/api-contracts` (например, `GetPaymentHistoryRequest`, `GetPaymentHistoryResponse`). Не описывайте структуры вручную.

2. Соберите объект запроса в соответствии с контрактом (поля `pagination`, `filters` и т.д. по proto). Используйте сгенерированные enum для полей-перечислений (например, `PaymentStatus.PENDING`).

3. Вызовите метод: методы сервиса возвращают **Promise\<ResponseType\>**:

   ```typescript
   const response = await client.getPaymentHistory(request);
   ```

   Убедитесь, что `response` типизирован как соответствующий Response-тип.

4. В Angular-сервисе оберните вызов в Observable с помощью `from()` из RxJS и при необходимости примените `map()` для преобразования ответа в UI-модель:
   ```typescript
   return from(this.client.getPaymentHistory(buildRequest(query))).pipe(
     map((response) => this.toTransactionHistoryPage(response)),
   );
   ```

**Контроль:** Запрос выполняется без ошибок типизации; тип ответа выводится автоматически.

---

### Шаг 3. Работа с enum

**Задание:** Использовать сгенерированные enum при формировании запроса и при маппинге ответа в UI-модель.

**Порядок выполнения:**

1. Импортируйте нужные enum из `@notary-portal/api-contracts` (например, `PaymentStatus`, `PaymentType`).

2. При построении запроса подставляйте значения enum, а не строки:

   ```typescript
   statuses: query.status ? [PaymentStatus.COMPLETED] : [],
   ```

3. В маппере ответа → UI делайте явное преобразование через `switch`/`case`, обрабатывая все нужные значения enum, включая при необходимости `UNSPECIFIED`.

**Контроль:** В коде не используются «магические» строки или числа для статусов/типов; везде используются константы из api-contracts.

---

### Шаг 4. Работа с полями типа Timestamp

**Задание:** Корректно преобразовывать поля `google.protobuf.Timestamp` при отображении в UI и при формировании запроса (границы дат).

**Порядок выполнения:**

1. Импортируйте из `@bufbuild/protobuf/wkt` функции `timestampDate` и `timestampFromDate`.

2. **Ответ → UI:** для поля с датой (например, `payment.paymentDate`) получите объект `Date` и при необходимости приведите к строке:

   ```typescript
   paymentDate: payment.paymentDate
     ? timestampDate(payment.paymentDate).toISOString()
     : '',
   ```

3. **UI → запрос:** для передачи границ даты в запрос создайте значение Timestamp из даты:
   ```typescript
   return timestampFromDate(new Date(`${value}T00:00:00.000Z`));
   ```

**Контроль:** Даты отображаются и отправляются без искажений; типы остаются совместимыми с контрактом.

---

### Шаг 5. Нормализация опциональных строк

**Задание:** При маппинге полей-строк из ответа в UI-модель учитывать пустые и пробельные строки (в proto строки не nullable).

**Порядок выполнения:**

1. Реализуйте вспомогательную функцию (или используйте из эталона), например:

   ```typescript
   function normalizeOptionalString(value: string): string | null {
     const normalized = value.trim();
     return normalized ? normalized : null;
   }
   ```

2. При маппинге полей, которые в UI должны быть «опциональными», используйте эту функцию вместо прямого присвоения строки из ответа.

**Контроль:** Пустые и пробельные значения не попадают в UI как есть; в модели используются `null` или пустая строка по конвенции проекта.

---

### Шаг 6. Организация слоя доступа к API

**Задание:** Вынести вызовы API в отдельный фасад-сервис и выполнять маппинг DTO → UI только в нём.

**Порядок выполнения:**

1. Создайте (или используйте существующий) сервис (в Angular — класс с декоратором `@Injectable`), который будет единственной точкой обращения к выбранному API-сервису.

2. В этом сервисе: создайте один экземпляр RPC-клиента (поле класса); реализуйте методы, каждый из которых вызывает один или несколько RPC-методов и возвращает уже UI-модель (или Observable её).

3. В компонентах используйте только методы этого фасад-сервиса; не вызывайте RPC-клиент и не маппите DTO в компонентах.

**Контроль:** Компоненты не импортируют типы Request/Response из api-contracts; все вызовы идут через фасад-сервис.

---

### Шаг 7. Обработка ошибок

**Задание:** Обрабатывать ошибки вызовов через тип ConnectError.

**Порядок выполнения:**

1. Импортируйте `ConnectError` из `@connectrpc/connect`.

2. В методе, возвращающем Observable, добавьте обработку ошибки через `catchError` (или обрабатывайте ошибку в подписчике при использовании Promise). При необходимости извлеките код и сообщение из `ConnectError` для отображения пользователю.

3. (По желанию.) Опишите, как в дальнейшем можно добавить HTTP interceptor для подстановки токена авторизации в заголовки запросов к API.

**Контроль:** При недоступности API или ошибке сервера приложение не падает; ошибка обрабатывается и при необходимости отображается пользователю.

## 8. Чек-лист выполнения работы

- [ ] Импорт сервиса и типов только из `@notary-portal/api-contracts`.
- [ ] Клиент создаётся через `createClient(Service, createConnectTransport({ baseUrl }))`, один экземпляр на сервис.
- [ ] Все запросы и ответы типизированы типами из api-contracts (Request/Response).
- [ ] Реализована обработка ошибок (ConnectError).
- [ ] Даты обрабатываются через `timestampDate` и `timestampFromDate` из `@bufbuild/protobuf/wkt`.
- [ ] Enum берутся из api-contracts; маппинг UI ↔ API выполняется явно (switch/case или аналог).
- [ ] Вызовы API инкапсулированы в фасад-сервисе; маппинг DTO → UI выполняется в сервисе, а не в компонентах.
