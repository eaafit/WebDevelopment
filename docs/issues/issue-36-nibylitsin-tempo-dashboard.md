## Описание

В issue 34 в проект была добавлена трассировка через **OpenTelemetry** и **Grafana Tempo**. В issue 35 основные бизнес-процессы API получили понятную разметку: `notary.operation`, `notary.entity`, `notary.actor.role`, `notary.result` и дополнительные доменные атрибуты.

Теперь нужно сделать отдельный дашборд в Grafana для просмотра этих трассировок. Он должен помочь быстро понять, какие бизнес-операции выполнялись, где были ошибки и в какие трассировки нужно провалиться для подробного разбора.

Дашборд не заменяет Grafana Explore. Он должен быть удобной стартовой точкой: открыть Grafana, выбрать дашборд по Tempo и сразу увидеть состояние основных бизнес-процессов.

## Текущая основа

В проекте уже есть:

- локальная Grafana через Docker Compose;
- Tempo как источник данных Grafana;
- отправка трассировок из API в Tempo через OTLP;
- связь логов с трассировками через `traceId` и `spanId`;
- разметка основных бизнес-процессов API стабильными атрибутами;
- автоматическая загрузка Grafana dashboards из JSON-файлов.

Новый дашборд нужно добавить в существующую схему provisioning, рядом с другими дашбордами Grafana.

## Что нужно реализовать

Добавить дашборд Grafana для бизнес-трассировок Tempo.

На дашборде должны быть фильтры, которые помогают смотреть трассировки по смыслу:

- бизнес-операция;
- сущность;
- роль пользователя;
- результат выполнения;
- временной диапазон.

Основные панели должны показывать:

- последние бизнес-трассировки;
- ошибки бизнес-операций;
- операции по группам: заявки, авторизация, платежи, документы, уведомления, рассылки, аудит, Bitrix, заказы, отчёты и профиль пользователя;
- медленные или проблемные операции, если это поддерживается текущей конфигурацией Tempo;
- быстрый переход от найденной трассировки к подробному просмотру в Tempo.

Если для части метрик требуется дополнительная настройка Tempo metrics-generator, её не нужно делать в этой задаче без необходимости. В первой версии дашборд должен работать на текущей локальной инфраструктуре.

## Сценарий просмотра

Преподаватель или разработчик открывает Grafana, выбирает дашборд по Tempo и видит, какие бизнес-процессы прошли через API.

Для демонстрации достаточно выполнить несколько действий в приложении:

- авторизоваться;
- создать или изменить заявку;
- открыть документ;
- получить уведомления;
- выполнить платежный сценарий или тестовый webhook;
- открыть аудит или список заказов.

После этого на дашборде должны появиться соответствующие бизнес-трассировки. По ним должно быть понятно, какая операция выполнялась, с каким результатом завершилась и где смотреть подробный trace.

## Связь с логами

Логи остаются в Loki и используются для текстовой диагностики.

Tempo показывает цепочку выполнения запроса. Дашборд должен помогать перейти к нужной трассировке, а связь с логами сохраняется через `traceId` и `spanId`.

В панели и фильтры дашборда не должны попадать персональные данные, полные идентификаторы, токены, адреса, имена файлов, ключи S3 и другие нестабильные значения.

## Проверка результата

Результат нужно проверить локально:

- поднять Grafana и Tempo через Docker Compose;
- запустить API с включённой отправкой трассировок;
- выполнить несколько основных операций приложения;
- открыть новый дашборд в Grafana;
- убедиться, что он загружается автоматически;
- проверить фильтры по бизнес-операции, сущности, роли и результату;
- открыть несколько трассировок из дашборда;
- убедиться, что дашборд не требует ручной настройки после перезапуска Grafana.
