## Описание

Внедрить **операционное** (техническое) логирование для API и веб-клиента: JSON-логи, идентификатор запроса, уровни логирования, политика по PII/секретам.

> **Не путать с продуктовым аудитом:** [issue-22](issue-22-nibylitsin-audit-log.md) и [issue-23](issue-23-chernenko-logs-security.md) — это UI и доменные события (аудит в админке). Данное issue — инфраструктурные логи процесса (ошибки, запросы, диагностика), они **не заменяют** таблицу аудита в БД.

## Бэкенд (NestJS, `apps/api`)

- [ ] Подключить **Pino** и интеграцию с NestJS (**nestjs-pino** или эквивалент): единый JSON в **stdout** в режиме, близком к боевому.
- [ ] Логировать входящие HTTP/Connect-запросы: метод, путь (без чувствительных query-параметров или с маскированием), код ответа, длительность.
- [ ] **Корреляция / идентификатор запроса:** принять заголовок (например `X-Request-Id` или `traceparent`), при отсутствии — сгенерировать; пробрасывать id во все логи запроса.
- [ ] Уровень логов из окружения (`LOG_LEVEL` / `NODE_ENV`); документировать переменные в описании PR или в комментарии к конфигу.
- [ ] **Не логировать** по умолчанию тела запросов/ответов, содержащие пароли, токены, платёжные и персональные данные; при необходимости — явный список разрешённых полей и маскирование.

## Фронтенд (Angular, `apps/web` и при необходимости общие библиотеки)

- [ ] Единый сервис логирования (обёртка): в локальном режиме — удобный вывод; в боевом режиме — минимальный шум, без утечки секретов из `localStorage`/токенов в сообщениях.
- [ ] Глобальный **`ErrorHandler`**: перехват необработанных ошибок, единый формат полей (`service`, `env`, опционально `userId` без PII).
- [ ] Опционально (по согласованию): пакетная отправка **только** error/warn на точку бэкенда с ограничением частоты и маскированием — не блокер для закрытия issue, если решено отложить.

## Технические ориентиры

- Зависимости: добавить в workspace (`package.json` / проект API) совместимые с Nest 11 версии `pino` / `nestjs-pino`.
- Точки интеграции: [`apps/api/src/main.ts`](../../apps/api/src/main.ts), корневой модуль приложения; bootstrap Angular — провайдер `ErrorHandler`.

## Критерии приёмки

- [ ] API в боевой сборке пишет структурированные JSON-строки в stdout с идентификатором запроса/корреляции на запрос.
- [ ] Запросы к API логируются с методом, путем, статусом и длительностью без сырых секретов.
- [ ] На фронте необработанные ошибки попадают в единый обработчик с согласованной схемой полей.
- [ ] Задокументированы переменные окружения для уровня логов (в PR или рядом с кодом конфигурации).
- [ ] Локально выполнена smoke-проверка: несколько запросов к API → в логах виден один и тот же request id на цепочку.

## Реализованный формат логов

Основная реализация использует Pino JSON-логи API в `stdout`. Логи входящих запросов содержат поля:

- `service`: всегда `api`
- `env`: значение `NODE_ENV` или `development`
- `level`: `trace`, `debug`, `info`, `warn`, `error`, `fatal` или `silent`
- `time`: метка времени в ISO-формате
- `requestId`: безопасный `X-Request-Id`, идентификатор трассировки из `traceparent` или сгенерированный UUID
- `method`: HTTP-метод
- `path`: путь URL без строки query-параметров
- `statusCode`: HTTP-код ответа
- `durationMs`: длительность запроса в миллисекундах

Уровнем логов бэкенда управляет `LOG_LEVEL`. Если переменная не задана, окружение с `NODE_ENV=production` использует `info`, а локальное окружение использует `debug`. Некорректные значения не ломают запуск приложения, а откатываются к резервному уровню.

Тела запросов, тела ответов и строки query-параметров по умолчанию не логируются. Чувствительные заголовки, включая `authorization`, `cookie`, `set-cookie`, `x-payment-webhook-secret` и `x-yookassa-signature`, маскируются, если попадают в логируемые метаданные.

Для issue 33 важно сохранять низкую кардинальность меток Loki: кандидатами на метки подходят `service`, `env` и `level`. Поле `requestId` должно оставаться полем для поиска и корреляции внутри полезной нагрузки JSON, а не меткой Loki.
