## Описание

Внедрить **операционное** (техническое) логирование для API и веб-клиента: JSON-логи, идентификатор запроса, уровни логирования, политика по PII/секретам.

> **Не путать с продуктовым аудитом:** [issue-22](issue-22-nibylitsin-audit-log.md) и [issue-23](issue-23-chernenko-logs-security.md) — это UI и доменные события (аудит в админке). Данное issue — инфраструктурные логи процесса (ошибки, запросы, диагностика), они **не заменяют** таблицу аудита в БД.

## Backend (NestJS, `apps/api`)

- [ ] Подключить **Pino** и интеграцию с NestJS (**nestjs-pino** или эквивалент): единый JSON на **stdout** в production-like режиме.
- [ ] Логировать входящие HTTP/Connect-запросы: метод, путь (без чувствительных query-параметров или с маскированием), код ответа, длительность.
- [ ] **Correlation / request id:** принять заголовок (например `X-Request-Id` или `traceparent`), при отсутствии — сгенерировать; пробрасывать id во все логи запроса.
- [ ] Уровень логов из окружения (`LOG_LEVEL` / `NODE_ENV`); документировать переменные в описании PR или в комментарии к конфигу.
- [ ] **Не логировать** по умолчанию тела запросов/ответов, содержащие пароли, токены, платёжные и персональные данные; при необходимости — явный allowlist полей и маскирование.

## Frontend (Angular, `apps/web` и при необходимости shared)

- [ ] Единый сервис логирования (обёртка): в dev — удобный вывод; в prod — минимальный шум, без утечки секретов из `localStorage`/токенов в сообщениях.
- [ ] Глобальный **`ErrorHandler`**: перехват необработанных ошибок, единый формат полей (`service`, `env`, опционально `userId` без PII).
- [ ] Опционально (по согласованию): батч-отправка **только** error/warn на backend-endpoint с rate limit и маскированием — не блокер для закрытия issue, если решено отложить.

## Технические ориентиры

- Зависимости: добавить в workspace (`package.json` / проект API) совместимые с Nest 11 версии `pino` / `nestjs-pino`.
- Точки интеграции: [`apps/api/src/main.ts`](../../apps/api/src/main.ts), корневой модуль приложения; bootstrap Angular — провайдер `ErrorHandler`.

## Acceptance criteria

- [ ] API в production-сборке пишет структурированные JSON-строки в stdout с полем request/correlation id на запрос.
- [ ] Запросы к API логируются с методом, путем, статусом и длительностью без сырых секретов.
- [ ] На фронте необработанные ошибки попадают в единый обработчик с согласованной схемой полей.
- [ ] Задокументированы переменные окружения для уровня логов (в PR или рядом с кодом конфигурации).
- [ ] Локально выполнена smoke-проверка: несколько запросов к API → в логах виден один и тот же request id на цепочку.
