## Описание

Детальный просмотр логов в контексте конкретного пользователя или заказа, а также отдельный раздел событий безопасности (входы, смены пароля, подозрительная активность).

> **Нибылицын Лукьян (issue-22):** общая лента аудита.  
> **Черненко Дмитрий (getpaintoo):** данное issue — логи по сущности + безопасность.

## Затронутые роли

- Администратор
- Нотариус (частично — только по своим заказам)

## Экраны / компоненты

- [ ] **Вкладка «Логи» в карточке пользователя** (`/admin/users/:id` → вкладка Logs) — хронология действий конкретного пользователя: входы, изменения профиля, платежи
- [ ] **Вкладка «Логи» в карточке заказа** (`/admin/orders/:id` → вкладка Logs) — история изменений статусов, комментарии, смена нотариуса, загрузка документов
- [ ] **Страница событий безопасности** (`/admin/monitoring/security`) — таблица: неудачные входы, смены пароля, подозрительные IP, блокировки; фильтры по IP, пользователю, дате
- [ ] **Badge «Подозрительная активность»** — выделение строк с высоким риском цветовой кодировкой

## Технические требования

- Маршруты: вкладки в `/admin/users/:id` и `/admin/orders/:id`; `/admin/monitoring/security`
- Защита: `roleGuard(UserRole.Admin)` из `@notary-portal/ui`
- RPC: `AuditService` с фильтром по `userId` / `orderId` / eventType=security
- Компонент логов — переиспользуемый: принимает `@Input() entityType` и `@Input() entityId`
- Координация с issue-22 (Нибылицын): общие компоненты строки лога / детали события

## Acceptance criteria

- [ ] В карточке пользователя вкладка «Логи» показывает только его события
- [ ] В карточке заказа вкладка «Логи» показывает историю изменений статуса
- [ ] Страница безопасности выделяет подозрительные события цветом
- [ ] Фильтрация по IP-адресу работает корректно
