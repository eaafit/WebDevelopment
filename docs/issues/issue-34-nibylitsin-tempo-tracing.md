## Описание

В проект необходимо добавить трассировку бэкенд-запросов через **OpenTelemetry** и **Grafana Tempo**.

Сейчас уже используется стек мониторинга: **Grafana**, **Loki**, **Prometheus** и структурированные JSON-логи API. Новая часть должна дополнить эту инфраструктуру и дать возможность просматривать путь выполнения одного конкретного запроса внутри API.

Основной сценарий для демонстрации — создание заявки на оценку.

## Текущая основа

В проекте уже есть:

- структурированные JSON-логи API;
- идентификатор запроса `requestId`;
- сбор технических логов через Loki;
- визуализация логов и метрик в Grafana;
- сбор метрик через Prometheus;
- локальная инфраструктура через Docker Compose.

Tempo добавляется как отдельный слой диагностики запросов через трассировку. Логи, метрики и аудит при этом сохраняют свои текущие роли.

## Что нужно реализовать

Добавить сервис **Tempo** в локальную Docker Compose-инфраструктуру и подключить его к Grafana как источник данных через существующий механизм автоматической настройки.

Подключить OpenTelemetry в API так, чтобы данные трассировки отправлялись в Tempo через OTLP. Инициализация трассировки должна выполняться до запуска NestJS-приложения, чтобы входящие запросы и внутренние операции API попадали в общий контекст выполнения.

В JSON-логи API необходимо добавить поля `traceId` и `spanId`. Это позволит связать записи в Loki с конкретной трассировкой в Tempo.

## Сценарий трассировки

Для первого примера используется создание заявки на оценку.

Трассировка должна отражать основные этапы обработки запроса:

- получение входящего API-запроса;
- выполнение `AssessmentService.createAssessment`;
- подготовку и проверку входных данных;
- создание записи через слой работы с данными;
- получение снимка созданной заявки;
- запись события аудита;
- создание уведомления;
- возврат результата клиенту.

По такой трассировке должно быть понятно, сколько занял весь запрос, какие внутренние этапы выполнялись и на каком этапе возникла ошибка.

## Связь с логами

Логи остаются основным источником текстовой диагностики: сообщения, ошибки, предупреждения и служебный контекст запроса.

Трассировка добавляет представление цепочки выполнения одного запроса. Для связи между логами и трассировкой используются поля `requestId`, `traceId` и `spanId`.

Высококардинальные значения, такие как `requestId`, `traceId`, `spanId` и идентификаторы сущностей, не должны выноситься в метки Loki. Они должны оставаться полями JSON-записи.

## Проверка результата

Результат нужно проверить локально:

- как поднять Grafana, Loki, Prometheus и Tempo;
- как запустить API с включённой трассировкой;
- как выполнить тестовое создание заявки;
- как найти трассировку в Grafana Tempo;
- как найти связанные логи в Loki по `requestId` или `traceId`.
