## Описание

Настроить цепочку **сбор → Loki → Grafana** для логов API (и при наличии — других сервисов), используя формат JSON из [issue-32](issue-32-nibylitsin-structured-logging.md).

> **Граница с продуктом:** экраны аудита в админке остаются в [issue-22](issue-22-nibylitsin-audit-log.md) / [issue-23](issue-23-chernenko-logs-security.md). Loki — для операционного просмотра и алертов по техническим логам.

## Зависимости

- **Блокер:** [issue-32](issue-32-nibylitsin-structured-logging.md) должен быть выполнен или согласован финальный формат JSON-полей (`level`, `time`, request id и т.д.), чтобы конфиг парсинга не ломался при мерже.

## Инфраструктура

- [ ] Добавить воспроизводимый пример развёртывания: **`docker-compose`** (или фрагмент под существующий деплой) с компонентами:
  - **Grafana Loki** (хранилище логов),
  - **Grafana** (визуализация),
  - агент сбора: **Promtail** или **Grafana Alloy** (чтение stdout/файлов контейнера с API).
- [ ] Конфиг агента: pipeline под **JSON**-строки из stdout API; при необходимости `json`/`cri` pipeline stages в документации Loki/Promtail.
- [ ] Метки (labels) для фильтрации: минимум `job`, `app`, `environment` (или эквивалент в терминах Loki без кардинальности взрыва — не использовать высококардинальные поля как labels).

## Grafana

- [ ] Подключить Loki как datasource в Grafana.
- [ ] Минимальная наблюдаемость: сохранённые запросы в Explore и/или простой дашборд: поток логов, фильтр по уровню (`error`), фильтр по **request id** (поле из issue-32).
- [ ] Опционально: одна панель или запрос, согласованный с существующими метриками Prometheus (`/metrics` в API) — без дублирования всей аналитики.

## Acceptance criteria

- [ ] После поднятия compose (или целевого окружения) логи API **видны** в Grafana (Explore) с предсказуемыми метками.
- [ ] По полю correlation/request id можно найти все строки одного запроса.
- [ ] В репозитории или в теле PR есть **краткая пошаговая инструкция**: как поднять стек и открыть логи (достаточно для повторения другим разработчиком).
- [ ] Обновление [developer-setup.md](../developer-setup.md) — по решению команды; если не трогаем файл, инструкция обязательно остаётся в описании PR или в `README` рядом с compose (если добавлен отдельный файл).
