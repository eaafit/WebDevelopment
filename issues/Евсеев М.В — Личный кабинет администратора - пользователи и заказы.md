## Контекст
- **Раздел:** `/admin/users` — управление пользователями  
- **Роли:** Администратор  
- **Связанные файлы:** `libs/web/admin/src/lib/`

## Задача
Реализовать CRUD-управление пользователями: список с фильтрами, карточка, редактирование, блокировка/разблокировка, просмотр заявок пользователя.

## API-контракты (`@notary-portal/api-contracts`)

| Метод | Запрос | Ответ | Где |
|---|---|---|---|
| `UserService.listUsers` | `{ pagination, roleFilter? }` | `{ users[], meta }` | Таблица пользователей |
| `UserService.getUserById` | `{ id }` | `{ user }` | Карточка пользователя |
| `UserService.updateProfile` | `{ userId, fullName, phoneNumber }` | `{ user }` | Редактирование |
| `AssessmentService.listAssessments` | `{ filters: { userId }, pagination }` | `{ assessments[], meta }` | Заявки пользователя |

> **Важно:** методы `deactivateUser`, `activateUser`, `changeUserRole` в proto **не реализованы** — реализовать как **заглушки** с оптимистичным обновлением UI и TODO-комментарием.

## Ключевые функции

### Список пользователей (`/admin/users`)
- Таблица: email, fullName, роль (читаемо: «Заявитель» / «Нотариус» / «Администратор»), статус (активен/заблокирован), дата регистрации
- **Фильтр по роли** — select из значений `UserRole` enum
- **Поиск** по email — клиентская фильтрация по загруженным данным
- Пагинация
- Кнопки в строке: «Открыть», «Заблокировать» / «Разблокировать» (mock)

### Карточка пользователя (`/admin/users/:id`)
- Блок данных: email, fullName, роль, телефон, статус, дата регистрации
- **Форма редактирования** (inline): `fullName`, `phoneNumber` → `UserService.updateProfile`
- **Блок «Заявки пользователя»:** таблица с пагинацией через `AssessmentService.listAssessments`
- Кнопки: «Заблокировать» / «Разблокировать» (mock), «Сменить роль» (mock — select + подтверждение)
