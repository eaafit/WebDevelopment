### 1.1. Диаграмма развертывания UML

**Требования к диаграмме:**

- Отобразить компоненты: Angular клиент для заявителя, API-сервер, PostgreSQL, файловое хранилище
- Показать взаимодействие с внешними сервисами: геокодирование для адресов, платежный шлюз
- Выделить модуль личного кабинета заявителя в общей архитектуре

**Пример компонентов:**

- Applicant Frontend → Assessment API → PostgreSQL + File Storage
- Внешние сервисы: Яндекс.Карты API (геокодирование), ЮKassa (оплата)

### 1.2. Описание развертывания на виртуальном сервере

**Что описать:**

1. **Контейнеризация:** Docker Compose с отдельными сервисами
2. **Переменные окружения:**
   - `ASSESSMENT_API_URL` для доступа к API заявок
   - `FILE_STORAGE_URL` для загрузки документов
   - `GEOCODING_API_KEY` для сервиса геокодирования
3. **Порты:** 4200 (frontend) → 3000 (API) → 5432 (PostgreSQL)
4. **Хранение файлов:** S3-совместимое хранилище для документов
5. **Мониторинг:** логи действий заявителя в системе

**Требования:**

- Applicant Dashboard → Assessment Service → PostgreSQL
- Document Upload Service → File Storage (S3/MinIO)
- Показать взаимодействие с Notification Service для уведомлений о статусах

### 2.3. API-методы и endpoints

**Список обязательных API-методов:**

| Метод | Endpoint                                    | Описание                | Параметры                   | Ответ                   |
| ----- | ------------------------------------------- | ----------------------- | --------------------------- | ----------------------- |
| GET   | `/api/applicant/assessments`                | Список заявок заявителя | `page`, `limit`, `status`   | `Assessment[]`          |
| POST  | `/api/applicant/assessments`                | Создать заявку          | `AssessmentCreateDto`       | `Assessment`            |
| GET   | `/api/applicant/assessments/{id}`           | Получить заявку         | `id`                        | `AssessmentWithDetails` |
| PUT   | `/api/applicant/assessments/{id}`           | Обновить заявку         | `id`, `AssessmentUpdateDto` | `Assessment`            |
| POST  | `/api/applicant/assessments/{id}/documents` | Загрузить документ      | `id`, `file`, `category`    | `Document`              |
| GET   | `/api/applicant/assessments/{id}/documents` | Список документов       | `id`                        | `Document[]`            |
| POST  | `/api/applicant/assessments/{id}/comments`  | Добавить комментарий    | `id`, `CommentDto`          | `Comment`               |

**DTO модели:**

- `AssessmentCreateDto`: address, description, realEstateObjectData
- `AssessmentUpdateDto`: description, contactInfo
- `CommentDto`: text, isInternal (true/false)

## Раздел 3. UI компонента (≈5 стр.)

### 3.1. Требования к интерфейсу

**Обязательные экраны:**

1. **Дашборд заявителя** — общая статистика: кол-во заявок по статусам, последние действия
2. **Список заявок** — таблица с фильтрами по статусу, дате, поиском по адресу
3. **Карточка заявки** — детальная информация: статус, история, документы, комментарии
4. **Загрузка документов** — форма для прикрепления файлов к заявке

**Функциональные требования:**

- Фильтрация заявок: по статусу (New, InProgress, Completed), дате создания
- Сортировка: по дате создания, статусу, адресу
- Поиск: по адресу, ID заявки
- Пагинация: по 10-20 заявок на странице
- Drag-and-drop загрузка документов
- Предпросмотр загруженных файлов (PDF, изображения)

### 3.2. Бизнес-правила

**Для роли Заявитель:**

1. Может создавать не более 5 активных заявок одновременно
2. Может редактировать заявку только в статусе "New"
3. Может загружать документы только к своим заявкам
4. Видит только свои заявки и документы
5. Получает уведомления при смене статуса заявки

**Ограничения доступа:**

- Просмотр заявок: только свои
- Создание заявок: Заявитель
- Редактирование заявок: только в статусе "New"
- Загрузка документов: к своим заявкам в статусах "New", "Verified"

### 3.3. Скриншоты UI

**Требования к скриншотам:**

1. Скриншот дашборда с статистикой
2. Скриншот списка заявок с примененными фильтрами
3. Скриншот карточки заявки с документами и комментариями
4. Скриншот формы загрузки документов с предпросмотром

### 3.4. Руководство пользователя

**Пошаговая инструкция подачи заявки:**

1. **Шаг 1:** Создание заявки
   - Пользователь нажимает "Новая заявка"
   - Заполняет форму: адрес объекта, описание наследства
   - Система сохраняет черновик
2. **Шаг 2:** Загрузка документов
   - Пользователь прикрепляет сканы документов
   - Система проверяет форматы и размеры файлов
3. **Шаг 3:** Отправка на проверку
   - Пользователь нажимает "Отправить"
   - Система меняет статус на "Verified"
   - Начинается процесс оценки

**Типичные сценарии:**

- Сценарий 1: Подача новой заявки с документами
- Сценарий 2: Отслеживание статуса существующей заявки
- Сценарий 3: Добавление документов к заявке в процессе оценки
