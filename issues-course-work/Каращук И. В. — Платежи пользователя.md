## Раздел 1. Модель развертывания системы (≈2 стр.)

### 1.1. Диаграмма развертывания UML
**Требования к диаграмме:**
- Отобразить все компоненты системы с акцентом на платежный модуль
- Показать внешний платежный шлюз (ЮKassa, Сбербанк Онлайн, Тинькофф) как отдельный узел
- Указать протоколы взаимодействия: HTTPS для API, webhooks для уведомлений
- Выделить компоненты: Payment Service, Subscription Service, Notification Service

**Пример компонентов:**
- Payment Frontend → Payment API → PostgreSQL
- External Payment Gateway (ЮKassa) → Webhook Handler → Payment Service
- Notification Service → Email/SMS для отправки чеков

### 1.2. Описание развертывания на виртуальном сервере
**Что описать:**
1. **Контейнеризация:** отдельные сервисы для платежей и подписок
2. **Переменные окружения:**
   - `PAYMENT_GATEWAY_API_KEY` для доступа к платежному шлюзу
   - `PAYMENT_GATEWAY_SHOP_ID` идентификатор магазина
   - `WEBHOOK_SECRET` для верификации webhook-уведомлений
3. **Порты:** стандартные порты + порт для webhook-сервера (например, 3001)
4. **Безопасность:** SSL/TLS для передачи платежных данных
5. **Мониторинг:** метрики успешных/неуспешных платежей

## Раздел 2. Ваш компонент (≈4 стр.)

### 2.1. Диаграмма классов UML
**Требования:**
- Использовать сущности из `docs/entities.md`:
  - **Payment**: Id, UserId, Amount, Currency, Status, Gateway, TransactionId, CreatedAt, PaidAt
  - **TariffPlan**: Id, Name, Description, Price, DurationDays, Features, IsActive
  - **Subscription**: Id, UserId, TariffPlanId, StartDate, EndDate, Status, AutoRenew
  - **Promo**: Id, Code, DiscountType, DiscountValue, ValidFrom, ValidTo, UsageLimit
  - **User**: Id, Email, FullName, Balance
- Показать связи: User 1:N Payment, User 1:N Subscription, TariffPlan 1:N Subscription
- Указать enum-типы: PaymentStatus (Pending, Processing, Completed, Failed, Refunded), SubscriptionStatus (Active, Expired, Cancelled)

### 2.2. Диаграмма компонентов UML
**Требования:**
- Payment UI → Payment API → Payment Service → External Gateway
- Webhook Listener → Payment Processor → Database
- Subscription Manager → Renewal Service → Notification Service
- Показать flow: выбор тарифа → создание платежа → редирект на шлюз → обработка webhook

### 2.3. API-методы и endpoints
**Список обязательных API-методов:**

| Метод | Endpoint | Описание | Параметры | Ответ |
|-------|----------|----------|-----------|--------|
| GET | `/api/payments/tariffs` | Список тарифных планов | `activeOnly` | `TariffPlan[]` |
| POST | `/api/payments/initiate` | Инициировать платеж | `PaymentInitiateDto` | `PaymentSession` |
| GET | `/api/payments/{id}` | Получить информацию о платеже | `id` | `Payment` |
| POST | `/api/payments/{id}/confirm` | Подтвердить платеж | `id`, `confirmationData` | `Payment` |
| GET | `/api/payments/history` | История платежей пользователя | `page`, `limit`, `dateFrom`, `dateTo` | `Payment[]` |
| POST | `/api/payments/apply-promo` | Применить промокод | `PromoApplyDto` | `DiscountInfo` |
| GET | `/api/payments/subscriptions` | Активные подписки | - | `Subscription[]` |
| POST | `/api/webhook/payment` | Webhook от платежного шлюза | `WebhookPayload` | `{success: boolean}` |

**DTO модели:**
- `PaymentInitiateDto`: tariffPlanId, promoCode, paymentMethod
- `PromoApplyDto`: code, tariffPlanId
- `WebhookPayload`: event, data, signature

## Раздел 3. UI компонента (≈5 стр.)

### 3.1. Требования к интерфейсу
**Обязательные экраны:**
1. **Выбор тарифа** — карточки тарифных планов с ценами и фичами
2. **Оплата** — форма ввода платежных данных, применение промокода
3. **История платежей** — таблица с фильтрами по дате, статусу, сумме
4. **Детали платежа** — информация о конкретном платеже, возможность скачать чек
5. **Управление подписками** — список активных подписок, отмена автопродления

**Функциональные требования:**
- Валидация платежных данных (номер карты, срок действия, CVV)
- Расчет итоговой суммы с учетом промокода
- Отображение прогресса платежа (инициализация → редирект → подтверждение)
- Фильтрация истории: по статусу (успешные, неуспешные), периоду, сумме
- Сортировка: по дате, сумме, статусу
- Экспорт истории: CSV/PDF выгрузка
- Уведомления об успешной/неуспешной оплате

### 3.2. Бизнес-правила
**Для ролей Заявитель и Нотариус:**
1. Оплата доступна только для активных тарифных планов
2. Промокод можно применить только один раз на пользователя
3. Промокод имеет срок действия и лимит использований
4. Чек отправляется на email после успешной оплаты
5. История платежей хранится 5 лет для налоговой отчетности
6. Возврат средств возможен в течение 14 дней

**Ограничения доступа:**
- Просмотр тарифов: все пользователи
- Инициация платежа: Заявитель, Нотариус
- Просмотр истории платежей: только свои платежи
- Применение промокодов: все пользователи
- Управление подписками: только свои подписки

### 3.3. Скриншоты UI
**Требования к скриншотам:**
1. Скриншот страницы выбора тарифа с примененным промокодом
2. Скриншот формы оплаты с заполненными данными
3. Скриншот истории платежей с примененными фильтрами
4. Скриншот деталей платежа с информацией о транзакции
5. Скриншот успешной оплаты с номером чека

### 3.4. Руководство пользователя
**Пошаговая инструкция оплаты:**
1. **Шаг 1:** Выбор тарифа
   - Пользователь переходит в раздел "Тарифы"
   - Выбирает подходящий тарифный план
   - Вводит промокод (если есть)
   - Нажимает "Оплатить"
2. **Шаг 2:** Ввод платежных данных
   - Система перенаправляет на страницу оплаты
   - Пользователь вводит данные карты
   - Подтверждает оплату
3. **Шаг 3:** Подтверждение
   - Система обрабатывает платеж
   - При успехе показывает подтверждение
   - Отправляет чек на email
   - Активирует подписку

**Типичные сценарии:**
- Сценарий 1: Успешная оплата тарифа с промокодом
- Сценарий 2: Просмотр истории платежей за период
- Сценарий 3: Отмена автопродления подписки
- Сценарий 4: Возврат средств по неудовлетворительной услуге
