## Раздел 2. Ваш компонент (≈4 стр.)

### 2.1. Диаграмма классов UML
**Требования:**
- Использовать сущности из `docs/entities.md`:

  - **User**: Id, Email, FullName, Role
  - **AuditLog** (для истории изменений): Id, UserId, ActionType, EntityName, EntityId, Timestamp, Details
- Показать связи: User 1:N Assessment, Assessment → AuditLog (история статусов)
- Указать enum Status: New, Verified, InProgress, Completed, Cancelled

### 2.2. Диаграмма компонентов UML
**Требования:**
- Assessment History UI → Assessment History API → PostgreSQL
- Real-time Updates → WebSocket Server → Clients
- Notification System → Email/SMS/Push Services
- Показать flow: изменение статуса → запись в историю → уведомление пользователя

### 2.3. API-методы и endpoints
**Список обязательных API-методов:**

| Метод | Endpoint | Описание | Параметры | Ответ |
|-------|----------|----------|-----------|--------|
| GET | `/api/assessments/history` | История заказов пользователя | `page`, `limit`, `dateFrom`, `dateTo` | `AssessmentHistory[]` |
| GET | `/api/assessments/{id}/timeline` | Таймлайн изменений заявки | `id` | `StatusChange[]` |
| GET | `/api/assessments/{id}/status-changes` | История статусов заявки | `id` | `StatusChange[]` |
| POST | `/api/assessments/{id}/repeat` | Повторить заказ | `id` | `NewAssessment` |
| GET | `/api/assessments/filters` | Доступные фильтры | - | `FilterOptions` |
| GET | `/api/notifications` | Уведомления пользователя | `unreadOnly` | `Notification[]` |

**DTO модели:**
- `AssessmentHistory`: assessmentId, address, status, lastUpdated, estimatedValue
- `StatusChange`: oldStatus, newStatus, changedAt, changedBy, comment
- `FilterOptions`: statuses, dateRanges, assessmentTypes

## Раздел 3. UI компонента (≈5 стр.)

### 3.1. Требования к интерфейсу
**Обязательные экраны/компоненты:**
1. **Лента заказов** — карточки/таблица с краткой информацией о заявках
2. **Детальный таймлайн** — визуализация истории изменений статуса конкретной заявки
3. **Фильтры и поиск** — панель фильтров по статусу, дате, типу недвижимости
4. **Повтор заказа** — кнопка/форма для создания копии существующей заявки

**Функциональные требования:**
- Фильтрация: по статусу (мультивыбор), периоду (дата от-до), типу недвижимости
- Сортировка: по дате создания, дате обновления, статусу, стоимости
- Поиск: по адресу, ID заявки, описанию
- Пагинация: бесконечный скролл или постраничная навигация
- Real-time обновления: автоматическое обновление при изменении статусов
- Экспорт истории: CSV/PDF выгрузка отфильтрованных данных

### 3.2. Бизнес-правила
**Для ролей Заявитель и Нотариус:**
1. Заявитель видит только свои заявки
2. Нотариус видит заявки, взятые в работу или все (в зависимости от прав)
3. Повтор заказа доступен только для завершенных заявок
4. Уведомления показываются только для релевантных изменений
5. История хранится 3 года для аудита

**Ограничения доступа:**
- Просмотр истории: Заявитель (свои), Нотариус (назначенные/все)
- Фильтрация: все поля доступны в зависимости от роли
- Повтор заказа: Заявитель (свои завершенные), Нотариус (только просмотр)
- Экспорт данных: Заявитель (свои), Нотариус (назначенные)

### 3.3. Скриншоты UI
**Требования к скриншотам:**
1. Скриншот ленты заказов с примененными фильтрами
2. Скриншот таймлайна заявки с историей статусов
3. Скриншот панели фильтров с расширенными опциями
4. Скриншот уведомлений об изменении статуса
5. Скриншот диалога повторения заказа

### 3.4. Руководство пользователя
**Пошаговая инструкция отслеживания заявки:**
1. **Шаг 1:** Просмотр ленты заказов
   - Пользователь заходит в раздел "Мои заявки"
   - Система показывает последние заявки с текущими статусами
   - Пользователь применяет фильтры для поиска нужных
2. **Шаг 2:** Детальный просмотр
   - Пользователь кликает на заявку
   - Система показывает таймлайн с историей изменений
   - Отображаются все документы и комментарии
3. **Шаг 3:** Действия с заявкой
   - Для завершенных заявок доступна кнопка "Повторить"
   - Система создает копию с исходными параметрами
   - Пользователь может отредактировать данные перед отправкой

**Типичные сценарии:**
- Сценарий 1: Отслеживание статуса новой заявки
- Сценарий 2: Поиск старых заявок по фильтрам
- Сценарий 3: Повторение успешной заявки для нового объекта
- Сценарий 4: Работа с уведомлениями об изменениях
