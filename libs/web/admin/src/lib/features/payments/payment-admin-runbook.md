# Runbook: платежи, чеки, CSV и Robokassa в админке

Документ описывает ручную проверку раздела `Администратор / Платежи` и связанных серверных сценариев. Он нужен для учебного проекта, потому что за время разработки было несколько повторяющихся багов: таблица зависала при первой загрузке, CSV молча не скачивался, чеки отдавали `503 Service Unavailable`, а dropdown перекрывался таблицей.

## Быстрый маршрут проверки

| Шаг | Страница | Действие | Ожидаемый результат |
|---|---|---|---|
| 1 | `/admin/payments` | Открыть список без фильтров | Таблица загружается без бесконечного loader |
| 2 | `/admin/payments` | Изменить `На странице` | Выполняется новый запрос с выбранным `limit` |
| 3 | `/admin/payments` | Выбрать `Статус` | Dropdown не уходит под таблицу |
| 4 | `/admin/payments` | Нажать `Экспорт CSV` | Скачивается CSV текущей выборки |
| 5 | `/admin/payments` | Открыть детали платежа | Модалка показывает сумму, статус, чек |
| 6 | `/admin/payments` | Нажать иконку чека | Чек открывается во вкладке |
| 7 | `/admin/payments` | Нажать download у чека | Файл скачивается как HTML/PDF |
| 8 | `/admin/orders` | Открыть заявки | Нет `internal error` в UI |
| 9 | `/admin/orders` | Проверить dropdown `Нотариус` | Список открывается над таблицей |
| 10 | `/admin/monitoring` | Открыть audit events | Нет raw `internal error` |

## Проверка CSV

| Проверка | Что смотреть |
|---|---|
| Имя файла | `payments-YYYY-MM-DDTHH-mm-ss.csv` |
| Кодировка | Excel должен открыть русские заголовки без кракозябр |
| Разделитель | `;`, чтобы русская локаль Excel читала колонки |
| Кавычки | Значения с `"` должны экранироваться двойными кавычками |
| Пустые поля | Для отсутствующих связей пишется `—` |
| Сумма | Сумма форматируется с двумя знаками |
| Комиссия | Пустая комиссия превращается в `0,00` |
| Статус | Экспортируется человекочитаемый статус |
| Фильтры | CSV берет текущую выборку, а не весь backend |
| Ошибка скачивания | UI показывает сообщение в toolbar |

## Проверка чеков

| Ситуация | Backend статус | UI поведение |
|---|---:|---|
| Пользователь не авторизован | 401 | Показать ошибку авторизации |
| Нет прав на платеж | 403 | Показать ошибку доступа |
| Чек еще формируется | 409 | Попросить попробовать позже |
| Чек отсутствует | 404 | Показать, что чек не найден |
| Storage недоступен | fallback | Отдать HTML-чек из данных платежа |
| Storage недоступен для PDF | 503 | Показать ошибку storage |
| Чек найден в storage | 200 | Открыть или скачать файл |
| Файл без content-type | 200 | Для просмотра считать HTML |

## Receipt fallback

Fallback нужен для локальной разработки и seed-данных. В учебной среде объектное хранилище может быть не поднято, а платеж уже содержит `attachmentFileName` и `attachmentFileUrl`. В таком случае пользователь не должен видеть только `503`.

Правило:

| Источник | Приоритет |
|---|---:|
| S3/MinIO объект | 1 |
| HTML fallback из платежа | 2 |
| User-facing ошибка | 3 |

Fallback не используется для PDF, потому что PDF нельзя корректно восстановить из платежных полей. Для HTML-чека восстановление допустимо: шаблон содержит плательщика, дату, сумму, метод оплаты, описание подписки или оценки.

## Audit события

| Событие | Когда писать | Контекст |
|---|---|---|
| `payment.receipt.opened` | Чек успешно открыт из storage | fileName, contentType |
| `payment.receipt.opened` | Чек восстановлен fallback | fileName, contentType |
| `payment.receipt.failed` | Чек pending | failureReason `receipt_pending` |
| `payment.receipt.failed` | Чек failed | failureReason `receipt_failed` |
| `payment.receipt.failed` | Нет attachment | failureReason `receipt_missing` |
| `payment.receipt.attached` | Ручной PDF загружен | objectKey, fileName |

## Notifications

| Сценарий | Кому | Что важно |
|---|---|---|
| Платеж создан | Пользователь | Понятный заголовок и сумма |
| Платеж создан | Администраторы | ID платежа и тип операции |
| Robokassa result success | Пользователь | Оплата подтверждена |
| Robokassa result success | Администраторы | Можно сверить audit и уведомления |
| Robokassa result failed | Администраторы | Ошибка не должна ломать webhook |
| Receipt fallback | Администраторы | Желательно смотреть audit, не создавать шумное уведомление |

## Robokassa: локальная проверка

| Параметр | Пример | Комментарий |
|---|---|---|
| `MerchantLogin` | `Test1999` | Тестовый магазин |
| `OutSum` | `1500.00` | Сумма должна совпадать с подписью |
| `InvId` | payment id | Лучше использовать локальный payment id |
| `Description` | `Подписка Базовый` | Пользовательский текст |
| `SignatureValue` | md5/hash | Генерируется из паролей Robokassa |
| `IsTest` | `1` | Тестовый режим |

Для полноценной проверки result URL нужен публичный callback или туннель. Без публичного адреса можно проверить генерацию ссылки, локальное создание платежа, audit event и обработчик result через прямой POST на локальный API.

## Типовые ошибки

| Симптом | Вероятная причина | Что делать |
|---|---|---|
| `503 object storage unavailable` | MinIO/S3 не поднят | Проверить fallback HTML |
| Чек не скачивается | Нет token или receipt endpoint вернул ошибку | Смотреть toolbar и console warn |
| CSV логирует success, но файла нет | `URL.createObjectURL` или click заблокирован | Проверить user-facing `exportError` |
| Dropdown под таблицей | Низкий z-index | Проверить `.admin-select` и `.admin-select__menu` |
| Таблица не влезает | Слишком широкие дефолтные col width | Проверить `columnWidths` |
| Заявки открываются с 500 | Backend assessment RPC | UI не должен показывать raw internal error |
